# Week 05 — Multi-Agent(必要性证伪优先) + Chaos 测试

## 本周目标

**周一是决策门** —— 先用数据证明 ResearchAgent 真的需要 Multi-Agent;通过则做 Coordinator/Worker;不通过则改做"模型路由"(同样有价值,且诚实)。无论选哪条,周末都加 Chaos 测试验证错误边界。

> [!warning] 默认应该是"不需要"
> Anthropic / Claude Code 的实战经验:大多数任务 single agent 够用。Multi-Agent 是收益明确才上,**不是默认架构**。

> [!info] 文件锁等并发硬化挪到 W6
> 本周只在 Coordinator 调度层做"同文件写串行"逻辑;真正的文件锁、写冲突排队放 W6。

  - **如果 single agent**:Judge 分 ≥ 80 + 耗时 < 90s + token < 12000
    → **本周改做"模型路由"**(simple subtask 走 Haiku,复杂走 Opus,验证成本/质量权衡)
    → 跳到本文末尾的【替代路径】
  - **否则**:走 Multi-Agent,继续下面
- [ ] 把决策结论写到 `docs/decision_multiagent_or_routing.md`(数据 + 选择 + 理由)

### 晚:走 Multi-Agent 路径,做接口设计

- [ ] 角色定义

| 角色 | 职责 | 可见上下文 |
|---|---|---|
| Coordinator | 分解任务、分配 Worker、合成结果 | 完整对话历史 + Memory |
| Searcher Worker | 搜索特定主题,返回结构化结果 | 自包含任务 Prompt |
| Analyst Worker | 分析数据,生成洞察 | 自包含任务 Prompt |
| Writer Worker | 撰写特定章节 | 自包含任务 Prompt |

- [ ] **类型化通信协议**(Pydantic):每个 Worker 定义 `Input` / `Output`
  ```python
  class SearcherOutput(BaseModel):
      items: list[SearchItem]   # 同时是 AnalystInput.sources 的类型
  ```
- [ ] Coordinator System Prompt 骨架(写到 `prompts/coordinator.md`)

> [!info] 设计原则(来自 Claude Code 源码)
> 1. 任务分阶段:Research → Synthesis → Implementation → Verification
> 2. 并行是超能力:只读任务自由并行,写任务同文件集串行
> 3. Worker Prompt 必须自包含
> 4. 合成理解不可委托
> 5. 默认 Spawn,Continue 仅在高上下文重叠时使用

## 并发执行 + Chaos 测试(周五)

### 周五 05-22

- [ ] `coordination/concurrency.py`:asyncio 或 ThreadPoolExecutor 并发执行器
- [ ] 调度规则:只读 Worker 自由并行,写同章节的 Worker 在 Coordinator 内部排队(真正文件锁放 W6)
- [ ] Worker 超时:单个 Worker 超时不影响其他,Coordinator 收到超时信号决策(retry / skip / 改派)

### `evals/chaos.py` — 3 类故障注入(关键验证)

- [ ] **故障1 — LLM 返回非法 JSON**:Mock LLM 返回 `{"items": [...` 缺尾。验证 Schema 自纠 + Reflection 能恢复
- [ ] **故障2 — Tool 部分成功**:`web_search` 返回错误信息但不抛异常(`{"error": "rate limited"}` 包在正常返回里)。验证 in-task critic 能识别"结果不可用"
- [ ] **故障3 — Memory 文件被外部进程删除**:测试中间手动 `rm` 一个 memory .md。验证错误边界:警告但不阻塞主流程
- [ ] 通过标准:3 类故障各能优雅恢复或诚实失败,没有崩溃 / 死循环 / 静默错误
