# Week 02 — MVP · LLMClient · Reflection 雏形 · Eval Harness 工业化

## 本周目标

补齐工具集到 6 个,封装 `LLMClient` 抽象层,加入 **Reflection 雏形**(LLM 输出后自检),建立 **Eval Harness 工业化版本**(LLM-as-Judge + golden answer + Mock-First),完成 MVP,产出 **W2 baseline**(后续 4 周必须超越的对照)。

> [!warning] 本周三件事不能省
> 1. **LLMClient 抽象** — 后续切 provider 只改一个文件,W6 CostTracker 才能聚合
> 2. **Eval Harness 工业化** — 字符串包含判定不算 eval,必须有 LLM-as-Judge + golden + 难度分级
> 3. **W2 baseline 落地** — 没有 baseline,W3-W6 谁也不知道是真改进还是退化

## 阶段1 收尾:LLMClient · run_python · 高风险工具(周一~周三)

### 周一 04-27 — Provider 抽象 + run_python

- [ ] 抽 `core/llm_client.py` 薄接口(~50 行)
  - `LLMClient.complete(messages, tools, stream=True) -> Iterator[chunk]`
  - 内部封装 retry / 流式解析 / token 统计
  - **构造函数注入**:`LLMClient(provider="openai", api_key=..., base_url=...)`,可被 eval 替换为 `MockLLMClient`
  - **现在切 provider 只改一个文件**
- [ ] 实现 `run_python` 工具(`subprocess.run(["python", "-c", code], timeout=10)`),`needs_permission = "ask_user"`
  - 真正的沙箱化(memory cap / 禁网)放到 W6
- [ ] 把 W1 写的 `agent_loop.py` 改造为通过 `LLMClient` 调用,而非直接调 SDK

### 周二 04-28 — 并行 · Audit · Token Logger

- [ ] 工具并行调用:LLM 一次返回多个 `tool_calls` 时,用 `ThreadPoolExecutor` 并发,按 id 顺序合并
- [ ] `core/audit_log.py`:每次工具调用追加 `{ts, session_id, tool, args, result_preview, duration_ms}` 到 `~/.researchagent/sessions/<id>/tools.jsonl`
- [ ] **Token Logger**(关键基础设施):在 `LLMClient` 内统计每次调用的 input/output tokens + duration,落 `~/.researchagent/sessions/<id>/llm.jsonl` —— W6 CostTracker 直接聚合,不用重新埋点

### 周三 04-29 — 新工具 + Schema 自纠 + Reflection 雏形

- [ ] 新增 `query_database` 工具(SQLite 只读,SQL 仅允许 SELECT)
- [ ] 新增 `send_notification` 工具(本地 webhook 或 print 即可,`needs_permission = "ask_user"`)
- [ ] Schema 验证:`InputSchema(**args)` 抛 `ValidationError` 时,把 Pydantic 错误信息原文回给 LLM,验证 LLM 能修正参数重试
- [ ] **Reflection 雏形**(简化版,完整闭环放 W4)
  - 在 `agent_loop.py` 主循环输出最终回复前,加一步 `critic_pass(reply, original_query) -> bool`
  - critic 用同一个 LLM,prompt 大致:"判断下面的回答是否真的解决了用户的问题。只输出 YES 或 NO + 一句理由。"
  - 如果 NO:把 critic 反馈作为 system message 注入,**重答 1 次**(不超过 1 次,避免无限循环)
  - 通过测试:故意问 "讲讲机器学习" 这种含糊问题,Agent 能识别出"我没真回答,需要追问澄清",而不是糊弄过去

## Eval Harness 工业化(周六)

### 周六 05-02 — 这是后续 4 周的回归底座,必须扎实

- [ ] **`evals/cases.jsonl`**:本周至少 5 条用例,**带难度分级**
  ```jsonl
  {"id": "w2-001", "difficulty": "easy", "query": "搜索 Tavily 是什么", "expected_tools": ["web_search"], "judge_criteria": "回答必须解释 Tavily 是搜索 API,且来源可追溯"}
  {"id": "w2-002", "difficulty": "medium", "query": "比较 LangGraph 和 LangChain", "expected_tools": ["web_search"], "judge_criteria": "回答必须包含至少 3 个具体差异点,且不能混淆两个框架"}
  {"id": "w2-003", "difficulty": "hard", "query": "讲讲机器学习", "judge_criteria": "Agent 必须识别问题过于宽泛并追问澄清,而非糊弄"}
  ```
- [ ] **`evals/golden/w2-001.md`** 等:为关键 case 写一份"理想回答",用于精确对比
- [ ] **`evals/judge.py`** — LLM-as-Judge 实现
  - 输入:`(query, reply, judge_criteria, golden?)` 
  - 输出:`{score: 0-100, reasoning: str, pass: bool}` (pass = score >= 70)
  - 用一个独立 LLM 调用(不复用业务 LLM 的对话上下文)
- [ ] **`evals/runner.py`** — 支持四个 mode
  - `--mock`:用预录 LLM 响应跑(< 5s 全集),开发循环用
  - `--real`:真 LLM 跑
  - `--judge`:跑完用 LLM-as-Judge 打分
  - `--baseline wN`:写 `evals/baseline/wN.json`(每 case 的 token / 耗时 / judge 分)
- [ ] **Mock-First 实现**:简单方案 — 在 `LLMClient` 加 `replay_from_file` 模式,W2 周五跑过的 baseline 数据自动转成 mock fixture
- [ ] **跑 W2 baseline**:`python evals/baseline.py --week w2`,生成 `evals/baseline/w2.json`
